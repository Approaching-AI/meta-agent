# 2026-03-13

## 新增：标准化 session 结束流程

在方法论中加入了 session 结束流程的标准化定义，解决了之前"工作结束"只有一句简单 prompt 示例、没有正式方法论支撑的问题。

### 改动内容

1. **doc/methodology.md** — 在"工作流程"section 后新增「Session 结束流程」section，定义三步标准流程：总结写入 daily notes → commit & push → 判断是否需要后续 agent 接手并生成 handoff prompt。同时更新了"常用 prompt 示例"中的结束收尾示例。

2. **init.sh** — 在 CLAUDE.md 模板中加入 session 结束流程的可执行指令，确保每个使用 meta-agent 的项目的 agent 启动时就自带这个流程。增加了对 `doc/methodology.md` 的引用。

### 设计考量

- CLAUDE.md 模板放可执行指令（告诉 agent 做什么），methodology.md 放方法论解释（说明为什么这样做）
- handoff prompt 不重复 daily notes 和 doc 已有内容，因为后续 agent 在同一个上下文体中运行
- 明确了不生成 handoff prompt 的情况：任务完成、或需要人类介入

## 重构：将具体指令从 init.sh 模板移入 submodule 内的 CLAUDE.md

发现一个架构问题：init.sh 把 session 结束流程等具体指令直接写进了项目的 CLAUDE.md，导致现有项目更新 submodule 后无法获得新指令。

### 解决方案

- 新建 `meta-agent/CLAUDE.md`（submodule 内），放所有具体操作指令
- `init.sh` 生成的项目 CLAUDE.md 精简为只引用 `meta-agent/CLAUDE.md`，不再内联具体内容
- 这样现有项目只需 `git submodule update --remote meta-agent` 即可获得所有更新

### 遗留问题

- 已有项目的 CLAUDE.md 里还保留着旧版内联内容，需要手动清理一次（一次性迁移成本）

## 新增：Handoff 文件持久化机制

将 session 结束时的 handoff prompt 从"输出文本"改为"写入文件"，形成完整的 session 开始/结束闭环。

### 设计决策

- **文件位置**：`handoff/` 目录
- **文件命名**：`YYYY-MM-DD-<简短描述>.pending.md`，包含日期和任务描述，每个 prompt 一个独立文件
- **文件内容**：纯 prompt，不加 frontmatter 或其他格式
- **状态标记在文件名中**：`.pending.md` 表示待执行，`.done.md` 表示已完成。Agent 不需要读文件内容就能判断状态
- **完成标记时机**：不绑定特定流程阶段，agent 在工作过程中任何时候判断任务完成就 rename

### 新增 Session 开始流程

- 检查 `handoff/*.pending.md`，有则读取作为任务指引，无则等待用户指令
- **开放问题**：多个 pending 文件的处理策略待定，暂由 agent 自行判断

### 改动文件

- `CLAUDE.md` — 新增开始流程，更新结束流程，新增 handoff 完成标记
- `doc/methodology.md` — 同步以上所有改动，更新常用 prompt 示例
